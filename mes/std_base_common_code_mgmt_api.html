1. 베이스 URL·환경

환경 동작
Vite dev + VITE_MES_DB_API_BASE 비어 있음 base === '' → 요청은 /db/..., /func/... (상대). renderer/vite.config.ts의 server.proxy로 https://100.71.84.10:7443에 전달(CORS 회피).
프로덕션(Electron 빌드 등) 기본 https://100.71.84.10:7443 + path, 또는 VITE_MES_DB_API_BASE로 지정한 베이스.
절대 URL 예 https://100.71.84.10:7443/db/tb_cm_code?...

2. 엔드포인트 요약

구분 Method Path 용도
상단 Group Code 콤보 GET /func/fn_cm_code p1=GROUPCODE, p2=code_name
그리드·목록 GET /db/tb_cm_code 공통코드 행 목록
저장(신규/수정) POST /db/tb_cm_code JSON body (buildTbCmCodePostBody)
삭제 DELETE /db/tb_cm_code 쿼리 code, 선택 code_group

3. GET — /func/fn_cm_code (상단 Group Code 콤보)

3.1 호출 코드

  • 파일: fnCmCodeApi.ts — fetchFnCmCodeOptions(p1, p2?)
  • 화면: 마운트 시 fetchFnCmCodeOptions('GROUPCODE', 'code_name')

3.2 URL·쿼리

GET {base}/func/fn_cm_code?p1=GROUPCODE&p2=code_name

3.3 호출 시점·조건

시점 조건 비고
화면 최초 마운트 항상 1회 Promise.all로 그리드 GET과 동시 요청
이후 자동 재호출 없음 콤보 옵션은 마운트 결과를 state에 유지

3.4 응답 처리

  • JSON이 배열이 아니면 키 data / rows / result / items / records 중 첫 배열을 사용.
  • 행 → { value, label }: code/CODE를 value, code_name/CODE_NAME/codeName을 label(없으면 code).
  • 중복 value는 제거(첫 항목 유지).

3.5 응답·UI

  • 필터 PlainSelect: value = 그룹 코드, 표시 = label.
  • 상세 Group Code PlainSelect: 동일 옵션 + mergeLegacyGroupOption으로 DB에만 있는 그룹은 (현재값) 옵션 추가.

3.6 에러

원인 동작
res.ok === false throw new Error(\GET fn_cm_code ${status}: …`)`
화면 catch → 인라인 배너 조회 실패: … (그리드 GET과 같은 try이면 한 번의 실패로 둘 다 중단)

4. GET — /db/tb_cm_code (그리드)

4.1 호출 코드

  • 파일: tbCmCodeApi.ts — fetchTbCmCodes(query)
  • 쿼리 빌드: buildQuery — 항상 code_group__like, _orderBy=disp_seq; 선택적으로 code, code_name__like

4.2 URL·쿼리

전체(첫 화면):

GET {base}/db/tb_cm_code?code_group__like=&_orderBy=disp_seq

그룹 필터(예: DEPT):

GET {base}/db/tb_cm_code?code_group__like=DEPT&_orderBy=disp_seq

(code_group__like가 비어 있어도 키는 항상 전송됨.)

파라미터 전송 조건 화면에서의 의미
code_group__like 항상 생략 시 빈 문자열 → “전체”. Search 시 filterGroupCode.trim()
_orderBy 항상 disp_seq
code TbCmCodeQuery.code가 비어 있지 않을 때만 이 화면에서는 미사용
code_name__like trim 후 비어 있지 않을 때만 이 화면에서는 미사용

4.3 호출 시점·조건

시점 호출 인자 트리거
마운트 fetchTbCmCodes({ code_group__like: '' }) useEffect 1회 (Promise.all의 한쪽)
Toolbar Search / 조회줄 fetchTbCmCodes({ code_group__like: filterGroupCode.trim() }) runSearch
저장 성공 후 위와 동일(현재 필터) runSave → runSearch()
삭제 성공 후 위와 동일 executeDelete → runSearch()

4.4 응답 처리

  • 타입: TbCmCode[] — 필드는 아래 §7 참고.
  • 파싱: 래핑 객체의 배열 필드(data/rows/…) 지원.
  • 빈 배열이면 그리드 비움, 상세 초기화(applySelectionAfterLoad).

4.5 에러

원인 동작
res.ok === false throw new Error(\GET tb_cm_code ${status}: …`)`
화면 조회 실패: … 배너, loading 해제

5. POST — /db/tb_cm_code (저장)

5.1 호출 코드

  • 함수: postTbCmCode(body) — tbCmCodeApi.ts
  • 본문: buildTbCmCodePostBody — tbCmCodeMapper.ts

5.2 URL

POST {base}/db/tb_cm_code
Content-Type: application/json
Accept: application/json

5.3 클라이언트 검증(저장 전)

  • 기존 행: useDirtyCheck로 변경 없으면 저장 안 함 + 배너 “변경된 항목이 없습니다.”
  • code 비어 있으면 mapper에서 throw → “저장 실패: Code는 필수입니다.”

5.4 JSON 필드(매퍼 기준)

필드 출처·비고
code 필수, trim
code_name, code_group, attribute01…attribute05, disp_seq, description trim
activated Y / N (상세 Activity yes/no)
create_date, created_by 신규: 현재 시각 / ADMIN; 수정: 기존 행 유지
update_date 항상 현재 ISO 시각
updated_by 기존 유지(없으면 '')

5.5 호출 시점

  • Toolbar Save, 조회줄 저장 연동 시 runSave에서 1회.

5.6 에러

원인 동작
!res.ok throw new Error(\POST tb_cm_code ${status}: …`)`
화면 저장 실패: …

6. DELETE — /db/tb_cm_code

6.1 호출 코드

  • 함수: deleteTbCmCode({ code, codeGroup }) — tbCmCodeApi.ts

6.2 URL·쿼리

DELETE {base}/db/tb_cm_code?code={code}&code_group={codeGroup}
  • code_group은 detail.codeGroup trim 후 비어 있지 않을 때만 쿼리에 포함.

6.3 호출 시점·조건

시점 조건
인라인 삭제 확인 후 deleteConfirmCode 설정 뒤 확인 버튼
요청 전 신규 모드이거나 existingCode 없으면 “DB에 등록된 코드 행을…”
클라이언트 code 없으면 deleteTbCmCode에서 throw

6.4 에러

원인 동작
!res.ok throw new Error(\DELETE tb_cm_code ${status}: …`)`
화면 삭제 실패: …, 확인 상태 초기화

9. 화면 수준 동작·UX

  • 로딩: 조회 중 loading true → 조회줄에 “조회 중…” 표시.
  • 배너: 성공/실패 메시지 약 3.2초 후 자동 제거.
  • 정렬: MesDataGridPanel 내부 정렬 시 onSortChange → 상세·선택 초기화(데이터 재요청 없음).
  • 줄 선택: 그리드 클릭 시 onSelectRow로 CommonCodeDetail 구성.

10. 보완·주의점(제안)

  1. 마운트 시 콤보·그리드 결합 실패: Promise.all로 한쪽만 실패해도 전체 catch — 필요 시 분리해 그리드만 표시하거나 콤보만 재시도.
  2. TbCmCodeQuery.code / code_name__like: API는 지원하나 화면 미사용 — 단건 조회·이름 검색 UI가 생기면 연동 검토.
  3. POST 응답 본문: 현재 성공 여부만 ok로 판단; 서버가 검증 오류 메시지를 JSON으로 주면 파싱해 배너에 넣을 수 있음.
  4. DELETE PK 규약: 서버가 복합키를 code만으로는 부족하다면 code_group 필수 여부를 백엔드와 재확인.
  5. Activity 필드: 읽기 시 activated와 activity 혼용 매핑 — DB 스키마 단일화 시 mapper 단순화 가능.
  6. 오프라인·타임아웃: fetch 기본 타임아웃 없음 — 필요 시 AbortSignal·재시도 정책 추가 검토.
