std_base_user_mgmt — HTTP API·쿼리 정리

std_base_user_mgmt 화면에서 나가는 HTTP 호출은 모두 mesDbUrl()로 만든 URL이며, GET은 두 종류(사용자 테이블·공통코드 함수)이다.

1. 베이스 URL (mesDbUrl)

  • 파일: renderer/src/lib/mesDbBaseUrl.ts
  • 개발(DEV) + VITE_MES_DB_API_BASE 비어 있음 → 베이스 '' → 최종 URL은 /db/..., /func/... 상대 경로(Vite proxy로 백엔드 전달).
  • 프로덕션 또는 env 설정 시 → https://…:7443 등 + path.

2. GET — 사용자 목록 tb_cm_user

  • 함수: fetchTbCmUsers — renderer/src/lib/tbCmUserApi.ts
  • 형태: GET {base}/db/tb_cm_user?{쿼리스트링}

쿼리 파라미터

파라미터 의미 화면 조건
user_id__like 사용자 ID 부분 일치(서버 규약) 필터 사용자ID에 값이 있을 때만
user_name__like 사용자명 부분 일치 필터 사용자명에 값이 있을 때만
_orderBy 정렬 항상 user_id

사용 위치 (std_base_user_mgmt.tsx)

  1. 마운트 1회: fetchTbCmUsers({}) → 실질적으로 ?_orderBy=user_id 만.
  2. 조회(Search): 필터 trim 후 비어 있지 않을 때만 user_id__like / user_name__like 설정 후 fetchTbCmUsers(q).
  3. 저장 성공 후: runSearch() 재호출 → 당시 필터와 동일한 GET.

응답

JSON이 배열이 아니면 data / rows / result / items / records 중 배열 필드를 찾아 TbCmUser[]로 파싱.

3. GET — 공통코드 fn_cm_code (부서·직급)

  • 함수: fetchFnCmCodeOptions(p1) — renderer/src/lib/fnCmCodeApi.ts
  • URL:
    • GET {base}/func/fn_cm_code?p1=DEPT
    • GET {base}/func/fn_cm_code?p1=DUTY
  • 호출: 화면 마운트 시 Promise.all로 동시 2회.
  • 용도: 상세 PlainSelect 옵션; 그리드 부서·직급 열은 resolveFnCmCodeLabel로 명칭 표시(별도 GET 없음).

5. 요약 표

구분 GET path 쿼리 호출 시점
사용자 /db/tb_cm_user _orderBy=user_id + 선택 user_id__like, user_name__like 최초 마운트, Search, 저장/삭제 후 재조회
부서 /func/fn_cm_code p1=DEPT 마운트 1회
직급 /func/fn_cm_code p1=DUTY 마운트 1회

서버 쪽 파라미터 의미는 백엔드(DB 래퍼) 규약에 따른다.
