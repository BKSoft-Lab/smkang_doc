1. 베이스 URL·환경

환경 동작
Vite dev + VITE_MES_DB_API_BASE 비어 있음 base === '' → 요청은 /db/..., /func/... (상대). renderer/vite.config.ts의 server.proxy로 https://100.71.84.10:7443에 전달(CORS 회피).
프로덕션(Electron 빌드 등) 기본 https://100.71.84.10:7443 + path, 또는 VITE_MES_DB_API_BASE로 지정한 베이스.
절대 URL 예 https://100.71.84.10:7443/db/tb_cm_user?...

2. 엔드포인트 요약

구분 Method Path 용도

3. GET — /db/tb_cm_user

3.1 호출 코드

  • 파일: tbCmUserApi.ts — fetchTbCmUsers(query)

3.2 URL·쿼리

GET {base}/db/tb_cm_user?{쿼리스트링}

buildQuery가 항상 _orderBy=user_id를 붙인다.

파라미터 전송 조건 의미
user_id__like trim 후 비어 있지 않을 때 사용자 ID 부분 일치
user_name__like trim 후 비어 있지 않을 때 사용자명 부분 일치
_orderBy 항상 user_id

3.3 호출 시점·조건

시점 인자 트리거
화면 최초 마운트 {} useEffect 1회 → 실질적으로 ?_orderBy=user_id만
Toolbar Search / 조회줄 user_id__like, user_name__like (필터에 값 있을 때만) runSearch
저장 성공 후 Search와 동일(현재 필터) runSave → runSearch()
삭제 성공 후 동일 executeDelete → runSearch()

3.4 응답 처리

  • JSON이 배열이 아니면 키 data / rows / result / items / records 중 첫 배열을 TbCmUser[]로 사용.
  • 타입 TbCmUser: tbCmUserApi.ts 참고(아래 §6과 동일 필드명).

3.5 에러

원인 동작
res.ok === false throw new Error(\GET tb_cm_user ${status}: …`)`
화면 catch → 인라인 배너 조회 실패: … (flashSaveBanner('err', …))

4. GET — /func/fn_cm_code (부서·직급)

4.1 호출 코드

  • 파일: fnCmCodeApi.ts — fetchFnCmCodeOptions(p1)

4.2 URL

GET {base}/func/fn_cm_code?p1=DEPT
GET {base}/func/fn_cm_code?p1=DUTY

4.3 호출 시점

  • 화면 마운트 시 Promise.all로 동시 2회.
  • 이후 자동 재호출 없음.

4.4 응답·UI

  • 행 { code, code_name } → { value: code, label: code_name }.
  • 그리드 부서·직급 열: resolveFnCmCodeLabel로 명칭 표시.
  • 상세 PlainSelect: 코드 저장, 라벨 표시. 코드표에 없는 기존값은 (현재값) 옵션 병합(mergeLegacyCodeOption).

4.5 에러

원인 동작
res.ok === false throw new Error(\GET fn_cm_code ${status}: …`)`
화면 catch → 배너 공통코드(DEPT/DUTY) 로드 실패: …

5. POST — /db/tb_cm_user (저장)

5.1 호출 코드

  • 함수: postTbCmUser(body) — tbCmUserApi.ts
  • 본문 생성: buildTbCmUserPostBody — tbCmUserMapper.ts

5.2 호출 시점

  • Toolbar Save, 조회줄 저장 연동 시 runSave 성공 분기에서 1회.

5.3 클라이언트 검증(저장 전)

  • 신규: Password 비어 있거나 ****만이면 저장 거부.
  • 기존: 상세 변경·비밀번호 변경 없으면 저장 거부.
  • Password 변경 시 Pass 체크 입력·일치 필수.
  • buildTbCmUserPostBody 내부에서도 Password/Pass 체크 불일치 시 throw.

5.4 POST JSON 필드(매퍼 기준)

필드 출처·비고
user_id 상세 사용자ID (필수)
user_name 상세 사용자명
pass 평문 비밀번호(변경 시). 미변경 시 기존 existing.pass
user_pwd 비밀번호 변경 시 MD5(평문).toUpperCase(), 미변경 시 기존 user_pwd
dept, duty 상세 콤보(코드)
mobile 상세
beginning_employment_date 기존 행 값 또는 당일 YYYY-MM-DD
ip_address, user_initial 상세
resigned, activated 상세 셀렉트(기본 N / Y)
description 상세 textarea
attribute01 기존값 또는 기본 'Y'
create_date, created_by 신규 시 현재 시각·ADMIN 등
update_date 항상 현재 시각(ISO)
updated_by 기존 유지 또는 빈 문자열

미전송(그리드/상세에 있으나 매퍼·타입에 없음): telephone, address 등은 POST 본문에 포함되지 않음(현재 구현). DB에 컬럼이 있어도 이 화면 저장으로 갱신되지 않는다.

9. 보완·주의점(제안)

  1. PK 중복: user_id가 논리적 PK인데 동일 ID로 여러 행이 있으면 find로 첫 행만 편집·삭제 대상이 되어 데이터 정합이 깨질 수 있음. DB UNIQUE 제약·API upsert 규약 점검 권장.
  2. 비밀번호: pass 평문 저장·MD5만 사용은 보안 정책상 개선 필요(해시 알고리즘·서버 측만 저장 등).
  3. telephone / address / E-Mail: DB·UI·POST 일관성 정리 권장.
  4. GET …?_count=true: 일부 환경에서 500이 보고됨 — 백엔드 지원 여부 확인.
