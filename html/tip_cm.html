CM관련 정리

CM 에서 IP 잡는 부분을 CMTS 표현함
CM 의 시리얼 메세지..
Scanning DS Channel at 501000000 Hz...(from HCN Golden Frequency)..
Scanning DS Channel at 501000000 Hz...(from HCN Golden Frequency).. 나오면서 IP 잡기 위한 Scanning 함.
scanning 하는 대역대 및 우선 순위들이 있음.
→ Docsis 라인을 제거 했다 다시 연결시 IP 잡는 부분이 느린 이유 중 하나가 A주파수에서 IP를 잡았는데.. 라인을 빼면 cm은 다른 주파수로 scan 진행함..
IP 잡기위해.. Scan 정책에 의해 A 주파수와 먼 곳을 Scan 중이라면 다시 A 주파수까지 잡기까지 시간이 걸려 어떤때는 빠르게 IP를 잡지만 어떤때는 느리게 IP를 잡음.
관련 내용은 24.04월 초쯤 SKB 쪽에서 마지막 잡은 주파수를 중간중간 scan하도록 개선 수정 및 테스트 한 것으로 알고 있음. HCN은 특별히 이야기 없으므로 적용 안함.
→ 참고 사항,
SKB는 RF 케이블을 뺐다 연결시 IP 를 최대 5분 잡는 문제를 개선 요청받아 개선하였음 ( 망 재연결 속도 개선 SKB 1.1.4 버전으로 릴리즈 됨 )
CM Tdsg3 timeout 30sec로 변경, Fast Scanning 수행 중 Last happy frequency 스캔 시도 추가 하여 개선 하였다고 메일 받음.
채널 정보 ( EPG )를 반는 부분을 DSG SI 테이터 라고 함.
이 정보 확인은 히든 메뉴의 4. 기타 정보 → DSG SI 데이터 상태 정보 에서 숫자 정상 인지로 확인 가능
HCN은 일주일 단위로 새벽에 soft-reboot을 한다고 함.
특정 SO 에서 soft-reboot 후 EPG를 못 받는 문제 발생한 부분이 있어 soft-reboot 시 stb, cm 모두 reboot 하도록 작업했음 ( 정확히는 모르지만 stb만 reboot시 터널이 해제후 다시 터널을 생성해야하는데 간혈적으로 터널을 생성 못하는 문제가 있음.. 터널생성이 정상적으로 생성시 ok를 주고 받으면 될것 같지만 이런 인터페이스가 전혀 없음.... 하여 같이 reboot 하도록 작업함. )

최종 문서는 아니지만 개발 중 전달 받은 OID 정보 문서 첨부 함 ( .230621확인_20110518_STB MIB_OID_정보_v1.3_XCAS추가_가온_해제.xls )
SNMP 로 CM에서 Test 하는 방법은 아래 CM 명령어 참조
SNMP로 CM에 명령을 보내는 주체는 STB도 있지만, 서버에서도 보낼 수 있으므로,
만약 특정 이상 동작이 발생 했을 때 STB에서 보내는지도 확인해야 하지만 외부로부터 명령이 오지 않은지도 확인 필요함.
예) SKB의 CM Reset 명령은 서버에서도 옴 (가입/미 가입 설정 때)
HCN RMS 서버에서 셋탑의 모델 명 얻어 갈 때
HCN 은 부팅 초기에 특정 채널 주파수 8개를 locking 하여 TSID (필터링해서 얻는 PID 같음) 를 획득 하는 절차가 있음.
얻은 TSID 정보는 HCN 설정 메뉴에서 확인 가능 함.
이 정보로 VOD 서버를 분배한다고 함.
→ 파악한 원리와 HCN의 담당자 말을 유추해서 이해 한 것으는 VOD 시작시 TSID 확인하여 TSID 첫번째가 1111 이면 1번서버 접속하고
1번 서버가 번잡하면 서버 쪽에서 1111번 unlocking 되도록 한후 TSID 재스캔 명령을 STB에 보내면 STB는 다시 주파수 8개를 스캔하여 하여 TSID 가 1112 부터 갖고 있게하여 2번 서버 접속하게 분배 한다고 함.

CM 업데이트 경로 3가지
a. 펌웨어 안에포함됨 cm 버전 업데이트 ( 실행된 cm 버전보다 높을때 진행 됨 )
b. 사업자가 OTA 로 업데이트 시키는 경우 ( MW단에서 다운받아 지정된곳에 저장해 놓고, plapiapp-system pl_hcn notifycmupgrade 0 호출해주어 업데이트 진행 )
c. 히든메뉴 → 제조사메뉴 → SW UPGRADE → CM USB upgrade로 진행 ( 방법 안내 문서 참조 : HCN_HowTo_USB_cm_update.doc )
CM 업데이트 진행 시 CM은 SNMP로 업데이트 명령을 받으면 버전 비교 등 전혀 하지 않고 TFTP 접속하여 파일 있으면 댕겨 받고 없으면 바로 STB에 reboot 하라고 SNMP를 전송 후 CM도 리붓 함.
펌웨어에 포함된 CM 버전이 STB 설치되어 있는 펌웨어보다 높으면 자동업데이트 진행 되게 됨. 부팅후 Alti_SNMP 프로그램이 올라오면 진행함
